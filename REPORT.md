# Отчёт: подготовка данных для обучения Whisper (башкирский язык)

Дата: 2026-07-19.

**Важно:** само обучение (`whisper.py`) на момент отчёта **не запускалось** —
ни разу не скачивался базовый чекпоинт `openai/whisper-medium`, в кэше
Hugging Face (`~/.cache/huggingface/hub`) есть только модель форс-алайнера
(`MahmoudAshraf/mms-300m-1130-forced-aligner`). Ниже — то, что сделано на
пути к обучению, фактическое состояние данных на диске, и что нужно
доделать, прежде чем запускать `whisper.py`.

## 1. Источники данных

Корпус строится из трёх независимых источников речи:

1. **Аудиокниги и рассказы** из Telegram-канала `bashaudiokitap` —
   собираются `scrape_tg_books.py` → `books/` (583 mp3) + `books_metadata.csv`.
2. **Стихи в исполнении школьников** (записи с YouTube, ссылки — в
   `bashkir_poems_bashkir.csv`) — скачиваются `download_bashkir_audio.py`.
3. **Диалектологические полевые записи** в формате ELAN (`elang_data/`) —
   готовые аудио+разметка, не требуют отдельного скачивания.

Плюс должен быть официальный **Mozilla Common Voice (Bashkir)** — но на
момент отчёта он **не скачан** (см. раздел 5).

## 2. Шаг 1 — сбор аудиокниг (`scrape_tg_books.py` → `books/`)

Выполнено: в `books/` лежит 583 mp3-файла. Статистика по ним (`books/stats.csv`,
собрана на основе `books_metadata.csv`):

- 214 уникальных авторов (часть — с пометкой «Билгеһеҙ автор», если пост в
  канале не содержал явного автора);
- 572 уникальные пары автор/название;
- покрытие метаданными: автор указан у 515 из 712 записей `books_metadata.csv`,
  название — у 687 из 712 (поле `reader`, диктор, не заполнено ни разу —
  распознавание диктора из тегов/подписи не сработало).

Это сырой материал для аудиокниг; сам по себе он **не является** обучающим
корпусом — нужен текст для сопоставления (шаг 3) и форс-алайнмент (шаг 4).

## 3. Шаг 2 — сбор текстов для сопоставления

Для части аудиокниг разыскивались тексты произведений:

- `find_author_works.py` — поиск по авторам из `authors.csv` на
  `kitaptar.bashkort.org` → `author_works.csv` (161 найденная ссылка на книгу).
- `download_pdfs.py` — скачивание самих PDF по этим ссылкам → `pdf/`.
- `download_kitaptar_collections.py` — отдельно докачаны собрания сочинений
  («Әҫәрҙәр», «Һайланма») → `kitaptar_collections/`.
- `search_kitaptar.py` — сверка уже собранных аудиокниг (`books_metadata.csv`)
  с библиотекой, чтобы понять, для каких вообще существует текст-первоисточник
  (`kitaptar_search_results.csv`).

Результат — папки `pdf/` и `kitaptar_collections/` с PDF/DjVu, из которых
вручную выбирались тексты, реально совпадающие с конкретными аудиофайлами
(зафиксировано в `pdf_audio_matches.md`).

## 4. Шаг 3 — сопоставление аудио и текста, укладка в `pairs/` и `pairs_poems/`

Из всего массива в 583 аудиокниг и найденных PDF **вручную** отобрана и
разложена по структуре `<Автор>/<Произведение>/{audio,pdf}` только часть:

- **`pairs/`** (проза): 8 авторов, 13 произведений, 48 исходных mp3.
- **`pairs_poems/`** (стихи, включая докачанные `download_bashkir_audio.py`

Это значит, что в форс-алайнмент и, соответственно, в обучающий корпус
попало пока **48 из 583** аудиокниг из `books/` — основная часть `books/`
(без пары «текст найден и вручную сверен») в корпус ещё не включена. Это
главный резерв для расширения корпуса «книг».

Логи ручной чистки дублей авторов/названий папок (объединение вариантов
написания вроде «Мустай Карим» → «Мостай Кәрим») — `merge_log.txt`,
`merge_log2.txt`, `merge_titles_log.txt`, `merge_state.txt`, `inspect.txt`.

## 5. Шаг 4 — форс-алайнмент (`ctc-forced-aligner/`)

Для каждой пары `audio + текст` в `pairs/` и `pairs_poems/` прогонялся
сторонний инструмент `ctc-forced-aligner` (git-клон в корне проекта),

Использованная модель форс-алайнера уже в кэше HF
(`MahmoudAshraf/mms-300m-1130-forced-aligner`).

python pairs_to_common_voice.py --input-dir pairs_poems --output-dir common_voice_pairs_poems --speaker-mode filename

## 7. Официальный Mozilla Common Voice — не готов

`whisper.py` по умолчанию ожидает также каталог `common_voice` (единственный
источник с настоящими `train/dev/test`-сплитами, а не «вырезанным» eval).
На диске такого каталога нет — есть только осиротевший `train.tsv` в корне
проекта (119 133 строки в исходном формате Common Voice: `client_id, path,
sentence, ...`), без папки `clips/`. Файл, судя по всему, — остаток
незавершённого скачивания. Прежде чем обучать модель с включением
официального датасета, нужно повторно скачать его:

```
python download_common_voice_mdc.py            # через Mozilla Data Collective (нужен MDC_API_KEY)
# либо
python download_dataset.py                     # через Hugging Face datasets (нужен HF_TOKEN)
```

## 9. Блокеры перед запуском обучения

1. **Зависимости.** Доустановить недостающее:
   `pip install torchaudio accelerate jiwer evaluate matplotlib`.
2. **ffmpeg.** Не найден в `PATH` — нужен для декодирования mp3-клипов при
   обучении. Поставить и добавить в `PATH`, либо (как уже сделано в
   `download_bashkir_audio.py`) использовать портативный бинарник из
   `imageio-ffmpeg`, адаптировав под это `whisper.py`.
3. **Устройство обучения.** Установленный `torch` собран под Intel **XPU**,
   CUDA недоступна (`cuda.is_available() == False`). `whisper.py` жёстко
   прописывает CUDA-специфичные опции (`bf16=True`, `tf32=True`,
   `optim="adamw_torch_fused"` в `Seq2SeqTrainingArguments`) — их нужно
   будет адаптировать под XPU (или явно посчитать, что реалистично обучать
   `whisper-medium` на CPU — на практике непрактично).
4. **Официальный Common Voice** отсутствует (раздел 7) — либо скачать его,
   либо осознанно обучать только на трёх «доморощенных» корпусах
   (`common_voice_pairs`, `common_voice_pairs_poems`, `common_voice_elang`,
   в сумме ~60.5 часов).
5. **Неполное покрытие `books/`.** Только 48 из 583 аудиокниг прошли
   сопоставление с текстом и форс-алайнмент; остальные — потенциальный
   резерв для расширения `pairs/`, но требуют ручной работы (найти текст,
   сверить, прогнать `ctc-forced-aligner`).

`test.tsv`, которого пока ни у одного из трёх нет, так что финальная
метрика на held-out тесте на этом наборе данных получена не будет.

## 11. Обновление 2026-07-19 (вечер) — обучение всё же прошло, результат слабее AigizK

Несмотря на блокеры из раздела 9 (нет CUDA, `torch` собран под XPU, часть
зависимостей отсутствовала), `whisper.py` **где-то всё же был обучен до
конца** — не в этом окружении (здесь по-прежнему нет папки `whisper-ba/`,
чекпоинтов и `training_curve.png`, а `git log` не содержит коммитов с
артефактами обучения), а на другой машине. В корень проекта позже добавили
только итоговый `trainer_state.json`.

### Что показывает `trainer_state.json`

- `max_steps=10000`, эффективный `train_batch_size=4`. Обучение остановилось
  именно по лимиту шагов (`should_training_stop: true` на `global_step=10000`),
  а не по концу эпохи.
- `epoch` на шаге 10000 равен **0.229** — пройдена меньше четверти **одного**
  прохода по объединённому корпусу. Обратным счётом (10000 × 4 / 0.229) это
  ≈174 600 примеров за эпоху — сходится с оценкой «~175k examples» из
  докстринга самого `whisper.py`; похоже, в этот прогон в итоге включили и
  официальный `common_voice`, в отличие от состояния на момент написания
  разделов 1–10 этого отчёта.
- Траектория `eval_wer` (сырой WER от `evaluate.load("wer")` внутри
  `compute_metrics()` — без `normalize_text()`/lowercasing, то есть не
  напрямую сравним с нормализованными метриками ниже, но сопоставим по
  порядку величины):

  | step | eval_wer, % | step | eval_wer, % |
  |---:|---:|---:|---:|
  | 500 | 65.17 | 6000 | 40.30 |
  | 1000 | 54.60 | **6500** | **34.68 (best)** |
  | 1500 | 54.45 | 7000 | 37.35 |
  | 2000 | 50.71 | 7500 | 39.24 |
  | 2500 | 44.41 | 8000 | 36.18 |
  | 3000 | 42.61 | 8500 | 34.85 |
  | 3500 | 40.33 | 9000 | 34.98 |
  | 4000 | 40.91 | 9500 | 35.60 |
  | 4500 | 46.45 | 10000 | 36.28 (final) |
  | 5000 | 39.38 | | |
  | 5500 | 37.53 | | |

  `load_best_model_at_end=True` сохранил чекпоинт **step 6500** (65% бюджета
  шагов) как финальную модель `whisper-ba`. После него WER больше не
  улучшался, а колебался в диапазоне 34.7–39.2%, притом что `eval_loss`
  (teacher-forced, по token-level кросс-энтропии) всё это время продолжал
  медленно снижаться (0.233 → 0.209). Расхождение между падающим `eval_loss`
  и переставшим падать `eval_wer` — типичный симптом того, что модель ещё
  учится предсказывать токен при доступе к правильному префиксу (teacher
  forcing), но её собственная авторегрессионная генерация (`generate()`, без
  teacher forcing — то, что видит пользователь и что меряет WER) уже не
  улучшается. Это ровно то поведение, которое видно в расшифровках
  `whisper_ba` из сверки ниже — зацикливания на повторе одного слова/фразы
  (`һуң һуң һуң...`, `ҡайҙаға ҡарап ҡайҙаға ҡарап...`).

### Независимая сверка на `common_voice_pairs_poems`

`whisper-ba` и публичный `AigizK/whisper-medium-ba` прогнаны через
`transcribe_evaluate_asr.py` на `common_voice_pairs_poems` (177 клипов;
результаты — `common_voice_pairs_poems_whisper_ba.csv` /
`common_voice_pairs_poems_aigizk.csv`, оба в корне проекта).

- В 47 из 177 клипов (27%) **обе** модели независимо получили WER ≥ 1.0 —
  признак не «модель плохая», а плохого форс-алайнмента исходного клипа.
  Эти строки убраны в `*_cleaned.csv` перед финальным сравнением.
- На оставшихся 130 клипах (`common_voice_pairs_poems_comparison.csv`):

  | Метрика | AigizK/whisper-medium-ba | whisper-ba (наш) |
  |---|---:|---:|
  | Corpus WER | 0.4282 | 0.4851 |
  | Corpus CER | 0.2170 | 0.2758 |
  | Corpus MER | 0.4211 | 0.4582 |
  | Corpus WIL | 0.6150 | 0.6582 |
  | Mean per-sample WER | 0.4588 | 0.5347 |

  AigizK лучше по всем метрикам. Порядок величины совпадает с внутренним
  `eval_wer` из обучения (34.68% на лучшем чекпоинте, без нормализации
  текста) — слабый результат воспроизводится на независимом корпусе, а не
  является артефактом методики оценки.

### Вероятные причины разрыва с AigizK

1. **Обучение остановилось на `max_steps=10000` — это 0.229 эпохи.** Модель
   ни разу не прошла весь объединённый корпус целиком. При фиксированном
   лимите шагов увеличение объёма данных не помогает и может вредить: каждый
   пример в среднем встречается модели реже. Чтобы «больше данных» реально
   давало эффект, `--max-steps` нужно поднимать пропорционально размеру
   корпуса (или переходить на обучение по числу эпох).
2. **`eval_wer` перестал улучшаться уже к шагу 6500** (65% бюджета), а к
   10000-му LR по расписанию (`warmup_steps=500`, дальше линейный спад к
   `max_steps`) почти обнулился — то есть даже в рамках имеющегося бюджета
   шагов последняя треть обучения была низкоэффективной.
3. **Шум в автоматической разметке.** Не менее 27% клипов
   `common_voice_pairs_poems` (см. выше) настолько плохо выровнены, что
   валят обе модели одновременно; если похожая доля брака есть в
   `common_voice_pairs`/`common_voice_elang` (основной объём корпуса, см.
   раздел 6), модель систематически учится на части примеров генерировать
   текст, не совпадающий по длине с реальным аудио — прямая причина
   зацикливаний при генерации.
4. **Малоресурсность башкирского в претрене Whisper.** `ba` подтверждённо
   есть в списке языков `WhisperTokenizer`, но в оригинальных 680k часах
   претрена Whisper башкирский был представлен минимально — качество
   файнтюна на таком слабом стартовом сигнале сильнее зависит от чистоты
   данных и глубины сходимости, чем для высокоресурсных языков, а именно
   этого (полного прохода по данным + чистых меток) прогону и не хватило.

### Что сделать в следующем прогоне

- Поднять `--max-steps` минимум до 2–3 полных эпох по факту используемого
  корпуса (при текущем эффективном batch size 4 это порядок сотен тысяч
  шагов) — либо считать бюджет явно через число эпох, а не фиксированное
  число шагов.
- Перед следующим прогоном применить к `common_voice_pairs` и
  `common_voice_elang` ту же проверку «WER обеих моделей на пересекающемся
  подмножестве» (или хотя бы разброс длины аудио/текста), что уже сделана
  для `pairs_poems`, и выбрасывать катастрофически невыровненные клипы из
  обучающего сплита, а не только из eval.
- Если после более длинного обучения зацикливания в генерации останутся —
  добавить `repetition_penalty`/`no_repeat_ngram_size` в вызов `generate()`
  (сейчас в `transcribe_evaluate_asr.py`/`transcribe_evaluate_whisper.py`
  декодирование чисто жадное, без защиты от повторов).
